# 嗨小问 CTC 可视化诊断说明

## 1. 这份文档解决什么问题

这份文档只回答两件事：

1. 当前 WebUI 里的诊断结果、可视化和各种统计分别是什么意思。
2. 如何借助这些可视化去判断 CTC 唤醒系统到底卡在什么地方，是声学输出、CTC 路径、beam search，还是最终阈值。

对应代码入口：

- WebUI: `examples/hi_xiaowen/s0/record_label_webui/app.py`
- 诊断后端: `examples/hi_xiaowen/s0/diagnose_wav.py`
- 单条 wav 推理: `examples/hi_xiaowen/s0/infer_wav.py`
- 训练期 CTC loss / prefix beam: `wekws/model/loss.py`


## 2. 先把训练和推理分开

### 2.1 训练时到底优化的是什么

训练期使用的是标准 `F.ctc_loss(...)`。

实现位置：

- `wekws/model/loss.py` 里的 `ctc_loss()`

核心过程：

1. 模型对每一帧输出一个 token 分布。
2. token 里包含：
   - `<blk>`: blank，表示这一帧“不输出实际字符”
   - 关键词字符，例如 `嗨`、`小`、`问`
   - 其他 token，例如 `<filler>`、`你`、`好` 等
3. CTC 不要求你给出每个字符精确落在哪一帧。
4. 它会把所有“能折叠成目标序列”的对齐路径概率加总，然后让这个总概率尽量大。

例如目标是 `嗨 小 问`，以下很多帧级路径都可能对应同一个标签：

- `<blk> 嗨 嗨 <blk> 小 <blk> 问 <blk>`
- `<blk> <blk> 嗨 <blk> 小 问 问 <blk>`

CTC 的规则是：

1. 先允许 blank。
2. 再允许重复。
3. 最后通过“去 blank + 合并相邻重复”折叠成标签序列。

所以：

- 训练时优化的是“目标序列的所有合法对齐路径总概率”
- 不是某一条唯一时间对齐路径

这也是为什么 CTC 很适合关键词检测：你不需要逐帧标注字符边界。


### 2.2 推理时看到的东西不是 loss 本身

WebUI 里的图和表，大部分都属于推理期诊断，不是训练期 loss 的直接展开。

推理期你看到的是：

1. 每帧后验分布
2. Greedy 解码结果
3. Prefix Beam Search 保留下来的路径
4. 在关键词约束下保留下来的路径
5. 完整关键词路径的乘积得分与阈值比较

也就是说：

- `CTC loss` 决定模型怎么学
- `greedy / beam / threshold` 决定模型学完以后怎么触发


## 3. 当前 WebUI 每个区域是什么意思

### 3.1 诊断结果总表

这个区块是把最关键的结论先收敛出来。

现在会显示：

- `整体判断`
- `Greedy 文本`
- `Beam Top1`
- `关键词 Beam 命中`
- `关键词 Beam 文本`
- `最强候选阈值差`

其中最重要的是 `整体判断`，它把问题归成三类：

1. `已触发`
2. `已形成完整关键词，但低于阈值`
3. `关键词约束 beam 里也没形成完整关键词`

这三类分别对应完全不同的调参方向。


### 3.2 波形与关键词候选时间位置

图上现在有两种区间：

- 橙色：`关键词候选区间`
- 绿色：`最终触发区间`

如何理解：

1. 只有橙色，没有绿色
   - 说明诊断 beam 能拼出一个完整关键词候选，但没通过最终触发条件
   - 可能是阈值太高，也可能是线上 beam/pruning 太严格
2. 绿色也有
   - 说明线上已经触发
3. 两种都没有
   - 说明当前可视化范围内甚至没有稳定候选

竖线和竖排 token 文本表示这条 beam 路径上每个 token 的时间位置。


### 3.3 CTC 原始输出：关键词相关 token 逐帧后验

这张曲线图是理解 CTC 最重要的图。

它画的是若干关键 token 在每一帧上的 posterior：

- `<blk>`
- `<filler>`
- 所有关键词相关 token，比如 `嗨`、`小`、`问`、`你`、`好`

怎么读：

1. 某个 token 的峰值越高，说明模型越相信这一帧是这个 token。
2. 如果目标关键词的几个 token 依次出现清晰峰值，CTC 就更容易形成完整路径。
3. 如果某个关键 token 峰值特别低，那么它通常就是漏检瓶颈。

例如 `嗨小问` 中：

- `嗨` 峰值很低
- `小` 和 `问` 峰值很高

那通常不是整个词都没学会，而是首 token 太弱，导致整条关键词路径很难被保留。


### 3.4 CTC 后验热力图

这张图和上一张是同一份数据的二维版本。

优点：

- 一眼能看出哪些 token 在哪些时间段亮起来
- 很适合对比 `<blk>` 是否长期压制所有关键词 token

重点观察：

1. 关键词 token 是否按顺序发亮
2. 有没有明显的错 token 抢占时间段
3. `<blk>` 是否在关键词段仍然过强


### 3.5 关键词约束 Beam Search 路径时间线

这是当前最接近“CTC 解码过程”的图。

每一行表示一条保留路径，横轴是时间。

颜色含义：

- 绿色：路径形成完整关键词，且 `candidate_score >= threshold`
- 橙色：路径形成完整关键词，但 `candidate_score < threshold`
- 蓝色：路径还没形成完整关键词

标签里的数值：

- `path_score`: beam 内部路径排序分数
- `cand`: 完整关键词命中后的候选得分
- `thr`: 当前关键词阈值

注意：

- `path_score` 用来排序路径
- `candidate_score` 才是最后拿去和阈值比较的值

两者不是同一个概念。


### 3.6 音频统计

这个区块回答的是“音频本身有没有明显质量问题”。

常看字段：

- `duration_sec`
- `peak`
- `rms`
- `sample_rate`

经验上：

1. `rms` 很低，可能说话太轻或离麦太远
2. `peak` 很低且整段都低，关键词 token 往往也会弱
3. 时长过短，首尾 token 可能被吃掉


### 3.7 平均帧 Top-K Token

这是整段音频所有帧的平均 posterior top-k。

它不是“真实 token 序列”，而是：

- 哪些 token 在整段音频里平均最强

用法：

1. 看 blank 是否极端统治
2. 看关键词 token 有没有整体进入前几名
3. 看是否有错误 token 长时间占优

如果 `<blk>` 长期接近 1，而关键词 token 平均值极低，通常说明模型整体不够相信这段音频包含关键词。


### 3.8 Greedy Token 序列

Greedy 的定义很简单：

1. 每一帧取最大概率 token
2. 去掉相邻重复
3. 去掉 blank

这只是最朴素的 CTC 解码，不做全局路径保留。

用法：

1. Greedy 已经接近正确关键词
   - 模型声学输出还行
   - 更可能卡在阈值或 beam 规则
2. Greedy 本身就缺字或错字
   - 更可能是声学模型没把 token 拉起来


### 3.9 关键词诊断

这是这次新增的区块，用来直接回答“到底卡在哪”。

每个关键词都会给出一行：

- `keyword`
- `tokens`
- `status`
- `candidate_score`
- `threshold`
- `score_margin`
- `weakest_token`
- `weakest_token_prob`
- `candidate_span`
- `best_text`

状态含义：

1. `已触发`
   - 完整关键词路径存在，且通过阈值
2. `已成词但低于阈值`
   - 关键词路径已经形成，但最终分数不够
3. `未形成完整关键词`
   - 在诊断 beam 里也没能拼出完整关键词
4. `已成词但无阈值`
   - 找到了完整路径，但当前没有阈值配置

最有用的字段通常是：

- `score_margin`
  - 小于 0 说明低于阈值
- `weakest_token`
  - 哪个 token 最拖后腿
- `weakest_token_prob`
  - 这个 token 有多弱


### 3.10 关键词约束 Beam Top-K

这是关键词约束 prefix beam 的前 K 条路径表。

关键词约束的含义不是“只能输出完整关键词”，而是：

- 允许的 token 集被限制在 `blank + 所有关联关键词 token`

所以它仍然可能输出：

- `小 问`
- `你 小 问`
- `小 小 问`

这些都是“在关键词 token 集里”的合法前缀，但不一定是完整关键词。


### 3.11 普通 Beam Top-K

这是不加关键词 token 约束的 prefix beam。

它用来看：

1. 如果不约束词表，模型本来最想输出什么
2. 错误 token 是否比目标 token 更占优
3. 约束 beam 是否只是把错误路径裁掉，而不是凭空造出正确路径


## 4. 当前这套系统里，真正决定是否触发的链路

### 4.1 线上推理链路

线上单条推理在 `infer_wav.py` 里。

步骤是：

1. wav -> fbank -> 模型 -> `probs`
2. 调 `ctc_prefix_beam_search(probs, utt_len, keywords_idxset)`
3. 在 beam 结果里找是否出现完整关键词子串
4. 如果出现，计算：

`candidate_score = sqrt(prod(token_probs_along_keyword_path))`

5. 再与阈值比较

只有 `candidate_score >= threshold` 才算触发。


### 4.2 线上推理和诊断页不是同一套搜索强度

这是理解错例时最容易忽略的点。

线上 `infer_wav.py` 调 beam 时使用的是 `ctc_prefix_beam_search` 默认参数：

- `score_beam_size=3`
- `path_beam_size=20`
- `prob_threshold=0.05`

而诊断页为了看全貌，使用了更宽松的设置：

- `score_beam_size=8`
- `path_beam_size=40`
- `prob_threshold=0.0`

这意味着：

1. 线上可能已经把弱 token 路径剪掉
2. 诊断页仍然能把这条路径“救出来”给你看

所以如果你看到：

- `infer_result.candidate_keyword = null`
- 但 `关键词诊断.status = 已成词但低于阈值`

这不表示“只是阈值太高”。

它更准确地表示：

- 诊断 beam 能找到该路径
- 线上真实 beam/pruning 没保住它
- 就算保住了，这条路径的 `candidate_score` 也仍然偏低

这类样本往往是：

- 弱 token
- beam/pruning
- 阈值

三者共同作用。


## 5. 一个真实错例怎么读

样本：

- `2026-03-18 14:11:37`
- wav: `20260318_141137_d9d771ca.wav`
- 人工标签: `嗨小问`
- 当前预测: `非唤醒词`

诊断页关键现象：

1. Greedy 只有 `小 问`
2. 普通 beam top1 也是 `小 问`
3. 诊断关键词 beam 可以找到 `嗨 小 问`
4. 但 `candidate_score = 0.0688 < threshold = 0.282`
5. `weakest_token = 嗨`
6. `weakest_token_prob = 0.0052`

这说明什么：

1. `小` 和 `问` 已经比较稳
2. `嗨` 非常弱，是整条路径的主瓶颈
3. 线上 beam 没把这条路径保住
4. 就算把路径保住了，当前阈值下也不会触发

所以这个错例不是“单纯阈值太高”。

更准确地说，是：

- 首 token `嗨` 声学后验太弱
- 导致路径容易被 prune
- 同时关键词乘积得分也被拉低


## 6. 如何用这些图做调参决策

### 6.1 目标是提高召回

先分情况。

#### 情况 A: 已形成完整关键词，但低于阈值

现象：

- `关键词诊断 = 已成词但低于阈值`
- `score_margin < 0`

优先动作：

1. 降阈值
2. 重算 DET，确认 FA/h 是否还能接受
3. 针对 `weakest_token` 补该关键词的数据覆盖

这是最典型的“召回靠阈值能换回来”的样本。


#### 情况 B: 诊断 beam 能成词，但线上 infer 没成词

现象：

- `infer_result.candidate_keyword = null`
- 但 `关键词诊断 = 已成词但低于阈值`

优先动作：

1. 检查 beam/pruning 参数是否过紧
2. 关注弱 token 峰值是否普遍偏低
3. 评估是否需要适当放宽：
   - `prob_threshold`
   - `score_beam_size`
   - `path_beam_size`

但要注意：

- 放宽 beam 通常会增加计算量和误触发风险
- 如果弱 token 本身极弱，只放 beam 往往治标不治本


#### 情况 C: 连诊断 beam 都没形成完整关键词

现象：

- `关键词诊断 = 未形成完整关键词`

优先动作：

1. 先别急着调阈值
2. 优先看 token 曲线和热力图
3. 检查是否存在：
   - 发音偏移
   - 首尾截断
   - 数据分布不匹配
   - 关键词 token 定义不合理
   - 模型容量不够

这类问题本质更偏“声学输出不对”，不是阈值问题。


### 6.2 目标是提高精确率

看的是另一类现象：

1. 非唤醒词样本也能形成完整关键词路径
2. `candidate_score` 还经常接近或超过阈值

这时优先动作是：

1. 提高阈值
2. 增加 hard negative
3. 看错误触发时最常见的错路径是什么
4. 看哪些 token 在非唤醒词上会虚高

如果错误触发总是集中在某个 token 组合上，说明不是整体阈值问题，而是词间可分性问题。


### 6.3 什么时候该优先补数据

满足下面任一条，优先补数据而不是先改 beam：

1. 某个关键词总是同一个 token 最弱
2. 这个弱 token 在不同说话人上都不稳定
3. Greedy 本身就经常缺这个 token
4. 热力图上该 token 从未形成清晰峰值

这说明模型并没有真正学稳这个 token。


### 6.4 什么时候该优先动 beam

满足下面组合时，优先检查 beam/pruning：

1. 关键词 token 曲线能看到顺序性峰值
2. 诊断 beam 能成词
3. 线上 infer 不能成词

这时问题更偏“路径保留策略”，不是纯声学问题。


## 7. 现在这套可视化够不够

结论：

- 对“看懂当前失败属于哪一类”已经基本够用
- 对“完整控制 CTC 行为”还不算终点，但已经能支撑大部分调参闭环

当前版本已经具备：

1. 帧级 token posterior
2. token 热力图
3. 关键词约束 beam 时间线
4. 关键词级诊断
5. 候选区间 vs 最终触发区间
6. weakest token 归因

还可以继续增强的方向：

1. 同一条样本下，线上 beam 与诊断 beam 的路径对比
2. 针对某个关键词的 token 乘积如何逐步形成
3. 多条错例聚合统计：
   - 哪个 token 最常成为 weakest token
   - 哪类错例最常是 pruning 问题
   - 哪类错例最常是 threshold 问题


## 8. 一句话版判断准则

如果你只记一条，就记这个：

1. `有完整关键词路径 + 分数差一点`：先看阈值和 weakest token
2. `诊断 beam 有，线上 infer 没有`：先看 pruning / beam，再看弱 token
3. `连诊断 beam 都没有`：先看声学输出和数据，不要先怪阈值
