# 20260428 199K 端侧量化内存与精度取舍

## 背景

本次排查对象是小雷 199K 当前候选：

- FP32：`exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10.pt`
- 原 native int8：`avg_10_stream_native_int8_calib200.tflite`
- 内存优化 native full-int8 IO：`avg_10_stream_native_fullint8_calib200.tflite`
- LiteRT/PT2E 对照：`avg_10_stream_litert_int8_pt2e.tflite`

触发问题：端侧反馈原 native int8 版本虽然参数仍是 199K，但内存占用超出约 11K；同时希望确认是否可以使用已成功部署的 229 模型同款 LiteRT/PT2E 导出方式。

## 核心直觉

参数量相同不代表端侧内存相同。TFLite 端侧内存主要还受以下因素影响：

- 输入/输出 tensor dtype，尤其 streaming cache 的 dtype。
- 中间 activation tensor 的 dtype 和生命周期。
- 导出器 lowering 后的 op 形态，例如 `DEPTHWISE_CONV_2D` 与 `CONV_2D`。
- runtime kernel 的 scratch buffer，这部分需要端侧 allocator 实测确认。

训练数据变化也不只是“样本变了”。训练完成后，权重、bias、head 判别边界、activation 分布、cache 状态分布都会变化。量化鲁棒性是数值分布属性，不由模型结构和参数量单独决定。

## 关键发现

### 1. 原 native int8 超内存的直接原因是 float32 IO

`avg_10_stream_native_int8_calib200.tflite` 的 sidecar 记录为：

- pipeline：`pytorch_onnx_onnx2tf`
- quant_mode：`int8_weights_float32_io`
- cache shape：`[1, 64, 11, 3]`

该模型输入输出仍是 float32：

- feats input：`[1, 1, 400] float32`
- cache input：`[1, 64, 11, 3] float32`
- out_cache output：`[1, 64, 11, 3] float32`
- logits output：`[1, 1, 20] float32`

cache 单边元素数是 `64 * 11 * 3 = 2112`。仅 cache input + output 从 int8 变 float32 就多：

```text
2112 * 2 * (4 - 1) = 12672 bytes = 12.4 KiB
```

这与端侧反馈“超约 11K”高度吻合。

### 2. native full-int8 IO 能解决内存问题且 test split 精度不掉

在 `torch2lite/quantize_tflite_native.py` 中增加 `--full_integer_io` 后，脚本会拷贝 onnx2tf 生成的 `*_full_integer_quant.tflite`，而不是默认的 `*_integer_quant.tflite`。

正式生成命令：

```bash
python -u torch2lite/quantize_tflite_native.py --config exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/config.yaml --checkpoint exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10.pt --output exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10_stream_native_fullint8_calib200.tflite --calib_data data_xlxl_0327_ctc_v1_clean/train/data.list --num_calib 200 --calib_backend onnxruntime --seed 20260323 --skip_onnx --dump_meta --full_integer_io
```

全量 streaming chunk300 评测结果：

| 模型 | 小雷小雷 | 小雷快拍 |
|---|---:|---:|
| FP32 `avg_10.pt` | 98.20%, fa/h 0.17 | 99.74%, fa/h 0.00 |
| native int8 float32 IO | 98.28%, fa/h 0.17 | 99.61%, fa/h 0.00 |
| native full-int8 IO | 98.28%, fa/h 0.17 | 99.61%, fa/h 0.00 |

结论：native full-int8 IO 在 test split 口径下没有引入额外精度回退。

### 3. flatbuffer tensor 体量对比

| 模型 | 文件大小 | I/O dtype | non-const tensor sum | I/O tensor sum |
|---|---:|---|---:|---:|
| 已部署 229 PT2E | 275,720 B | int8 | 28,152 B | 4,644 B |
| 小雷 native int8 float32 IO | 260,680 B | float32 | 47,240 B | 18,576 B |
| 小雷 native full-int8 IO | 260,128 B | int8 | 28,664 B | 4,644 B |
| 小雷 LiteRT/PT2E | 275,416 B | int8 | 28,088 B | 4,644 B |

native full-int8 IO 相对原 native int8 float32 IO：

```text
47,240 - 28,664 = 18,576 B
```

这已经覆盖端侧反馈的约 11K 超额。它相对已部署 229 PT2E 只多 `512 B` non-const tensor；但最终 arena 仍需端侧 allocator 实测，因为 scratch buffer 不完全体现在 flatbuffer tensor sum 中。

### 4. LiteRT/PT2E 对小雷当前模型精度明显回退

为了验证“是否可以直接使用 229 同款导出方法”，对小雷 `avg_10.pt` 跑了 LiteRT/PT2E：

```bash
python -u torch2lite/export_streaming_litert_tflite.py --config exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/config.yaml --checkpoint exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10.pt --output exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10_stream_litert_int8_pt2e.tflite --quant_mode int8_pt2e --chunk_frames 1 --calib_data data_xlxl_0327_ctc_v1_clean/train/data.list --num_calib 200 --seed 20260323 --dump_meta
```

结果：

| 模型 | 小雷小雷 | 小雷快拍 |
|---|---:|---:|
| native full-int8 IO | 98.28%, FRR 1.72%, fa/h 0.17 | 99.61%, FRR 0.39%, fa/h 0.00 |
| LiteRT/PT2E | 89.73%, FRR 10.27%, fa/h 0.17 | 97.91%, FRR 2.09%, fa/h 0.00 |

结论：小雷当前 `avg_10.pt` 不能使用 PT2E 作为部署候选，尤其“小雷小雷”明显崩。

### 5. 复跑 229 PT2E 证明导出方法本身没错

对 229 原 FP32 checkpoint 重新执行同款 PT2E 导出：

```bash
python -u torch2lite/export_streaming_litert_tflite.py --config exp/fsmn_ctc_distill_mini_align_20_test2/config.yaml --checkpoint exp/fsmn_ctc_distill_mini_align_20_test2/229.pt --output exp/fsmn_ctc_distill_mini_align_20_test2/229_stream_litert_int8_pt2e_verify.tflite --quant_mode int8_pt2e --chunk_frames 1 --calib_data data/train/data.list --num_calib 200 --seed 20260323 --dump_meta
```

新导出的 verify 文件与原已部署 229 文件 hash 完全一致：

```text
229_stream_litert_int8.tflite             5cebba4f505713114a16a5e419d984fa3ea408bf72f65851c5f5b12a18b606cf
229_stream_litert_int8_pt2e_verify.tflite 5cebba4f505713114a16a5e419d984fa3ea408bf72f65851c5f5b12a18b606cf
```

用当前脚本全量评测 73,459 条 test split：

| 模型 | 嗨小问 | 你好问问 |
|---|---:|---:|
| 229 PT2E verify | 97.65%, FRR 2.35%, fa/h 0.99 | 97.94%, FRR 2.06%, fa/h 0.51 |
| 229 原历史目录 | 98.05%, FRR 1.95%, fa/h 0.99 | 98.05%, FRR 1.95%, fa/h 0.53 |

当前脚本复跑和历史目录有轻微差异，可能来自评测脚本/后处理口径变化；但关键结论不变：229 使用 PT2E 仍是 97%+，没有出现小雷的 89% 级别回退。

## 为什么“只是训练数据变了”也会导致 PT2E 失效

PT2E 不是只量化权重；它会更激进地插入 activation/cache 的 quantize/dequantize。对 streaming FSMN 来说，cache 每帧都会输出并作为下一帧输入，量化误差会沿时间传递。

训练数据变更后，以下分布都会变化：

- 每层 linear/affine 权重与 bias。
- FSMN memory 层输出范围。
- head 对 token 和 blank 的边界。
- 小雷小雷 hard positive replay 后的 token 峰值位置与幅度。
- streaming cache 的动态范围和长尾样本。

CTC keyword 检测对 token 峰值非常敏感。若首个“小”或中间“雷”的峰值被 PT2E 量化压低，FRR 会显著上升。229 对 PT2E 鲁棒，只能说明 229 训练后的分布更适合这套 observer/scale；不能推出小雷当前模型也适合。

## 导出器差异说明

FSMN 训练代码里的 memory 层本身是 depthwise 形式：

```python
self.conv_left = nn.Conv2d(self.dim, self.dim, [lorder, 1], groups=self.dim, bias=False)
```

但是不同导出器会选择不同 TFLite lowering：

- LiteRT/PT2E：识别为 `DEPTHWISE_CONV_2D`。
- ONNX -> onnx2tf：经过 ONNX Conv group、layout 转换、slice/concat/cache 图处理后，lower 成 `CONV_2D`。

这不是训练结构变了，而是 compiler/exporter 的 op 选择不同。端侧内存最终还要结合目标 runtime 的 kernel scratch buffer 实测。

## Depthwise 后处理优化

端侧反馈 native full-int8 仍比 229 多约 4K。重新拆分后发现，flatbuffer 静态 tensor 并不能解释这 4K：

| 模型 | op 形态 | non-const tensor | IO tensor |
|---|---|---:|---:|
| 229 PT2E | `DEPTHWISE_CONV_2D:6` | 28,152 B | 4,644 B |
| 小雷 native full-int8 | `CONV_2D:6` | 28,664 B | 4,644 B |

两者 non-const 只差 `512 B`，因此 4K 更像是端侧普通 `CONV_2D` kernel scratch，而不是模型参数或 TFLite tensor arena。

onnx2tf 生成的 6 个 `CONV_2D` 满足：

- input channel = output channel = 64。
- weight shape 为 `[64, H, 1, 1]`。
- 每个输出通道只看 1 个输入通道，等价于 PyTorch 的 `groups=dim` depthwise conv。

因此新增后处理脚本把这些 `CONV_2D` 无损改写成 `DEPTHWISE_CONV_2D`：

```bash
python torch2lite/rewrite_group_conv_to_depthwise.py --input exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10_stream_native_fullint8_calib200.tflite --output exp/fsmn_ctc_xlxl_distill_199k_hardpos_replay_head_only_from119_v1/avg_10_stream_native_fullint8_calib200_depthwise.tflite --min-converted 6
```

后处理验证：

- 转换 `6` 个 op：`CONV_2D -> DEPTHWISE_CONV_2D`。
- 随机输入下 Python TFLite 对比原模型：`out_cache` 和 `logits` 最大绝对差都是 `0.0`。
- flatbuffer tensor 体量不变，目标是减少端侧 depthwise kernel 相对 generic Conv2D 的 scratch。

完整 test split 评测结果：

| 模型 | 小雷小雷 | 小雷快拍 |
|---|---:|---:|
| native full-int8 | 98.28%, FRR 1.72%, fa/h 0.17 | 99.61%, FRR 0.39%, fa/h 0.00 |
| native full-int8 depthwise | 98.28%, FRR 1.72%, fa/h 0.17 | 99.61%, FRR 0.39%, fa/h 0.00 |

结论：depthwise 后处理不改变精度，应作为当前首选端侧内存优化候选；最终是否吃掉端侧多出的约 4K，需要目标 TFLM/LiteRT allocator 实测。

## 当前结论

1. 小雷当前部署候选应使用：

```text
avg_10_stream_native_fullint8_calib200_depthwise.tflite
```

2. 不建议使用：

```text
avg_10_stream_litert_int8_pt2e.tflite
```

原因是 PT2E 对当前小雷模型精度明显回退。

3. 原 native int8 float32 IO 精度可用，但端侧内存不合适。

4. native full-int8 IO + depthwise 后处理是当前最平衡方案：test split 精度与原 native full-int8 持平，op 形态更接近已部署 229，预期能减少端侧普通 Conv2D scratch。

## 复用建议

- 新模型做端侧量化时，不要只看参数量和文件大小，必须检查 TFLite I/O dtype、non-const tensor sum、op_counts。
- 对 streaming 模型，优先关注 cache input/output dtype；cache 从 float32 改 int8 通常能显著降低 arena 压力。
- PT2E 结果不能从旧模型外推到新训练分支；每个 checkpoint 都需要独立评估。
- 如果 native full-int8 IO 精度不掉，应优先尝试 depthwise 后处理，再交给端侧 allocator 实测 arena。
- 若端侧仍超内存，再考虑进一步减少 cache shape、FSMN order、proj_dim，或针对 onnx2tf 图做 op/scratch 级别优化。
