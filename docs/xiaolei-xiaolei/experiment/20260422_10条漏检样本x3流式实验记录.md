# 20260422 10条漏检样本 x3 流式实验记录

## 1. 实验目的

前面已经确认：

- `distill 199K 119.pt` 在单条 hard case 上常把 `小雷小雷` 压缩成 `雷小雷`、`小雷` 或 `小小雷`
- 单条 repeated hard case `x3` 也已经看到：老师能稳定多次触发，学生容易漏第一段

这次补的是一组更系统的验证：

- 把 `logit_compare_小雷小雷_test_119` 里挑出的 10 条“老师收、学生漏”样本
- 全部做成 `x3` 连续拼接
- 再同时跑老师 / 学生的 streaming 行为

要回答的问题是：

- 旧学生 `119.pt` 对这批典型漏检样本，是否能在重复拼接后稳定做到 `3/3` 检出
- 还是会出现稳定的“漏第一段”模式

## 2. 输入与输出

输入样本来源：

- `examples/hi_xiaowen/s0/exp/fsmn_ctc_xlxl_distill_199k/analysis/logit_compare_小雷小雷_test_119/manifest.jsonl`

批量实验汇总：

- `examples/hi_xiaowen/s0/exp/fsmn_ctc_xlxl_distill_199k/analysis/repeated_hardcase_batch_x3_from_logit_compare_119.jsonl`

实验设置：

- 学生：`exp/fsmn_ctc_xlxl_distill_199k/119.pt`
- 老师：`exp/fsmn_ctc_xlxl_top20_weight_surgery/159.pt`
- 每条 wav 重复次数：`x3`
- streaming `chunk_ms = 300`

## 3. 结果总览

### 3.1 学生结果

- `0/10` 条能够正确做到 streaming `3/3`
- `9/10` 条只有 `2` 次触发
- 剩余 `1/10` 条虽然总共触发了 `3` 次，但分布不是 `0/1/2` 三段各一次，而是 `segment 1` 两次、`segment 2` 一次

更关键的是：

- `10/10` 条都漏掉了 `segment 0`
- 学生触发段位模式完全一致，都是只覆盖后两段

学生流式触发分布统计：

- `student_streaming_num_activations_x3 = 2`：`9` 条
- `student_streaming_num_activations_x3 = 3`：`1` 条
- 但 `student_streaming_segments_x3` 的唯一模式是：`(1, 2)`

### 3.2 老师结果

- `10/10` 条都覆盖了 `segment 0 / 1 / 2`
- `9/10` 条是标准 `3` 次触发
- `1/10` 条出现了额外重复触发，但仍然覆盖三段

老师流式触发分布统计：

- `teacher_streaming_num_activations_x3 = 3`：`9` 条
- `teacher_streaming_num_activations_x3 = 4`：`1` 条
- `teacher_streaming_segments_x3` 的唯一覆盖模式是：`(0, 1, 2)`

## 5. 结果的含义

### 5.1 这不是“偶尔漏一段”，而是稳定模式

这次最重要的结论不是简单的 `0/10`，而是：

- 这 10 条老师收、学生漏的代表性样本
- 在 `x3` 流式场景下
- 学生全部稳定漏掉第一段 `segment 0`

这说明旧学生 `119.pt` 的问题并不是随机波动，而是：

- 对这类 hard case，存在稳定的早期 prefix / 起步阶段失守

### 5.2 学生并不是“完全不会触发”

结果也说明：

- 学生对这批样本不是完全收不到
- 它能在 `segment 1`、`segment 2` 触发
- 但就是更容易丢掉最早出现的那一段

这比“学生整体召回低”更具体，说明问题更像：

- 长序列里的前缀起步能力不稳
- 第一段更容易被 blank 主导或路径压缩掉

而不是：

- 三段都一样差

### 5.3 非流式整段 triggered 不能代替 streaming 结论

这批 `x3` 实验里，学生 `10/10` 在非流式整段上都是 `triggered=true`。

但这并不代表部署场景里就真的能 `3/3` 检出。原因是：

- 非流式整段 CTC 会把整条长音频上的 token 路径拼成一个全局结果
- 它可以“整体触发”
- 但不等于流式推理时每一段都能独立站住

所以这次结果进一步说明：

- 对 `小雷小雷` 这种问题，不能只看 non-streaming `triggered`
- streaming repeated hard case 才更接近真实部署风险

### 5.4 老师的优势不是“每段都必须先有 filler”

前面的单条 repeated hard case 已经说明：

- 老师并不会在每段前都固定打一个 `<filler>`

这次批量 `x3` 的意义在于进一步确认：

- 老师真正稳定的是多段流式触发能力
- 学生真正缺的是第一段起步与前缀保持

所以这里不应把问题理解成：

- 学生少学了几个 `<filler>`

而更应理解成：

- 学生没有学好老师那种更稳的输出行为与路径保持能力

## 6. 对后续训练的直接启发

这批结果直接支持下面几条判断：

1. 后续训练 gate 里必须保留 repeated hard case 的 streaming 检查
2. 旧学生的主要问题不是“后两段也收不到”，而是“第一段稳定丢失”
3. 这支持继续走“让学生更贴近老师输出行为”的路线
4. output-level KL、blank/non-blank KD、以及后续 hard positive replay 都是合理方向
5. 不能只靠整段 non-streaming `triggered` 判断模型是否适合部署

## 7. 当前收口

这次批量 `x3` 实验把一个之前只在单条样本上看到的现象，扩大验证成了稳定模式：

- `distill 199K 119.pt`
- 对这 10 条典型 hard case
- 在流式 `x3` 条件下
- 表现为稳定漏掉第一段

因此后续如果新模型只是在原始 `test split` 上略有改善，但在这类 `x3` streaming gate 上仍然漏第一段，那么不能算真正解决问题。
