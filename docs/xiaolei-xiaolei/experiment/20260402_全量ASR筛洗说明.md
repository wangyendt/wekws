# 20260402 全量 ASR 筛洗说明

## 1. 这份文档是干什么的

这份文档面向“需要实际看数据、做清洗、安排人工复核”的同事，不是只给出几个统计数字，而是回答下面几个实际问题：

- 这次全量筛洗到底做了什么
- 生成了哪些文件，分别怎么看
- 哪些样本最值得先看
- 哪些更像标签错了，哪些更像切分不干净
- 如果要安排人工复核，优先级应该怎么排

本次筛洗对象为：

- `data_xlxl_0327_ctc_v1/dev/data.list`
- `data_xlxl_0327_ctc_v1/test/data.list`
- `data_xlxl_0327_ctc_v1/train/data.list`

处理方法是：用 `tools/screen_xiaolei_asr.py` 调 `paraformer-zh` 对每条音频做 ASR 转写，再比较“原始标注文本”和“ASR 文本”的关系，把样本分成 `clean / ambiguous / reject` 三类。

需要强调的是：

- 这一步不是自动改标签
- 这一步的价值是先把“最可能有问题的样本”从全量数据里筛出来
- 最终是否删样本、改标签、重切分，仍然需要人工复核决定

## 2. 一句话结论

如果只看最关键结论，可以先记住这 5 句：

1. 全量共筛了 `182168` 条样本，其中 `97.62%` 被归为 `clean`，`1.23%` 为 `ambiguous`，`1.15%` 为 `reject`。
2. 全量正样本共 `48212` 条，其中 `3984` 条不是 `clean_exact`，约占正样本的 `8.26%`。
3. 真正最像“正样本标错类”的，不是所有非 clean 样本，而是 `ambiguous_other_target + ambiguous_target_overlap` 这 `193` 条。
4. 数量最大的异常正样本是 `reject_positive_miss`，共 `1977` 条，这批需要靠人工听音进一步区分“标签错 / 切分错 / 录音差”。
5. `ambiguous_extra_context` 共 `1061` 条，这类多数不是标签错，更像切分边界偏长、前后带了额外词。

## 3. 这次跑完以后生成了什么文件

每个 split 都生成 5 个文件，目录分别在：

### 3.1 `asr_results.jsonl`

这是最重要的总表，后续绝大多数排查、导表、人工复核清单，都是从这里继续筛。

每一行对应一条原始样本，常用字段有：

- `key`: 样本 key
- `txt`: 原始标注文本
- `wav`: 音频路径
- `asr_text`: ASR 原始转写结果
- `asr_text_compact`: 归一化后的 ASR 文本
- `screen_status`: 机器筛洗状态
- `screen_bucket`: 三大桶之一，`clean / ambiguous / reject`
- `timestamp`: ASR 时间戳
- `asr_error`: 如果推理失败，这里会记录错误

如果同事只关心“哪些正样本可能标错了”，优先看这个文件里：

- `txt in {小 雷 小 雷, 小 雷 快 拍}`
- `screen_status != clean_exact`

### 3.2 `clean.data.list`

筛洗后暂时认为“问题不大，可以优先保留”的样本列表。

### 3.3 `ambiguous.data.list`

模型认为“有异常，但不适合直接删”的样本。一般需要人工复核。

### 3.4 `reject.data.list`

模型认为风险较高、应该优先复核或暂时剔除的样本。

### 3.5 `summary.json`

每个 split 的汇总统计文件，主要看三类信息：

- `counts`: `clean / ambiguous / reject` 各有多少
- `status_counts`: 每种细分状态各有多少
- `error_count`: ASR 失败多少条

## 4. 机器筛洗状态怎么理解

下面这张表是整个数据清洗过程中最重要的解释表。

| 状态 | 含义 | 更像什么问题 | 建议 |
|---|---|---|---|
| `clean_exact` | ASR 与正样本标注完全一致 | 基本正常 | 直接保留 |
| `ambiguous_extra_context` | ASR 含目标词，但前后还有别的词 | 更像切分不干净 | 人工复核，不建议直接删 |
| `ambiguous_partial_keyword` | 只识别到部分关键词 | 发音不完整 / 噪声 / 切分偏差 | 人工复核 |
| `ambiguous_other_target` | 标的是一个目标词，但 ASR 更像另一个目标词 | 高度怀疑错类 | 高优先级复核 |
| `ambiguous_target_overlap` | ASR 中出现目标词混叠或重叠 | 高度怀疑错类 | 高优先级复核 |
| `reject_positive_miss` | 正样本 ASR 完全没打到目标词 | 可能标错 / 录音差 / 切分错 | 高优先级复核 |
| `clean_negative` | 负样本未识别出目标词 | 基本正常 | 直接保留 |
| `ambiguous_negative_partial_keyword` | 负样本里出现部分关键词痕迹 | 可能混入目标词边界 | 人工复核 |
| `ambiguous_negative_target_overlap` | 负样本和目标词有重叠 | 有一定风险 | 人工复核 |
| `reject_negative_exact_target` | 负样本直接命中目标词 | 很可能负样本标错 | 高优先级复核 |
| `reject_load_error` | 文件损坏、读取失败或推理异常 | 工程问题 | 单独排查 |

一个很实用的判断原则是：

- `other_target / target_overlap` 更像“标签错类”
- `extra_context` 更像“切分问题”
- `positive_miss` 需要听音才能进一步判断是哪一类问题

## 8. 同事最关心的：哪些最像“正样本标错了”

如果目标是尽快找出“最可能标错标签”的正样本，不建议从所有非 clean 样本开始一锅端，而建议按下面的优先级看。

### 8.1 第一优先级：`ambiguous_other_target` + `ambiguous_target_overlap`

这是最像“正样本错类”的一批。

典型模式是：

- 标注写的是 `小 雷 小 雷`
- ASR 更像 `小 雷 快 拍`
- 或 ASR 里同时混入了目标词重叠信息

这批数量不大，但信息密度最高，适合作为第一批人工复核清单。

样例：

- `20260311-101247057_zhangsan_m_3s_copy3`
  - 标注：`小 雷 小 雷`
  - ASR：`小 雷 快 拍`
  - 状态：`ambiguous_other_target`
- `20260311-144210669_zhangsan_m_1.7s_copy2`
  - 标注：`小 雷 小 雷`
  - ASR：`小 雷 快 拍`
  - 状态：`ambiguous_other_target`
- `20260311-145906866_zhangsan_m_5s_copy3`
  - 标注：`小 雷 小 雷`
  - ASR：`小 雷 快 拍`
  - 状态：`ambiguous_other_target`

### 8.2 第二优先级：`reject_positive_miss`

这批不是“必然标错”，但很值得优先人工听音。

常见原因有：

- 真实说话内容和标注不一致
- 录音质量差，ASR 没打准
- 切分截断，关键词没有完整保留
- 关键词太弱，被 ASR 打成近音词

样例：

- `XIAOLEI_XIAOLEI_20250205_130000_05_4_rep_4`
  - 标注：`小 雷 小 雷`
  - ASR：`晓 得 晓 得`
- `XIAOLEI_XIAOLEI_0057_3m_f_04_fast_05`
  - 标注：`小 雷 小 雷`
  - ASR：`晓 得 晓 得`
- `XIAOLEI_XIAOLEI_0283_3m_m_04_fast_01`
  - 标注：`小 雷 小 雷`
  - ASR：`晓 得 晓 得`

这批量很大，不建议直接全删，更合适的做法是继续细分：

- 先看 ASR 打成固定近音词的聚类
- 再看来源目录是否集中在某批采集
- 再结合时长、波形、听音判断是切分问题还是标签问题

### 8.3 第三优先级：`ambiguous_extra_context`

这类通常不是“标签错”，而是“切分不干净”。

常见形态例如：

- `嗯 小 雷 小 雷`
- `小 雷 小 雷 了`
- `小 雷 小 雷 快 拍`
- `说 啥 小 雷 小 雷`

这类样本怎么处理，要看训练目标：

- 如果要做严格唤醒词训练，建议重切分或人工裁边界
- 如果要增强真实场景鲁棒性，可以视情况保留一部分

## 9. 实际复核时建议怎么安排

为了让同事更容易落地，我建议不要直接把所有非 clean 样本一次性丢过去，而是拆成 3 批。

### 第一批：最可能标签错类的正样本

筛选条件：

- `txt in {小 雷 小 雷, 小 雷 快 拍}`
- `screen_status in {ambiguous_other_target, ambiguous_target_overlap}`

特点：

- 数量不大
- 价值最高
- 人工结论最稳定

### 第二批：ASR 完全没打中目标词的正样本

筛选条件：

- `txt in {小 雷 小 雷, 小 雷 快 拍}`
- `screen_status == reject_positive_miss`

特点：

- 数量最大
- 需要听音
- 这批最适合继续做二次聚类或人工分桶

### 第三批：切分边界问题样本

筛选条件：

- `txt in {小 雷 小 雷, 小 雷 快 拍}`
- `screen_status == ambiguous_extra_context`

特点：

- 更像切分问题，不是标签问题
- 后续应与“标签错”分开处理

## 11. 这份结果怎么用于训练集清洗

建议不要简单按 `ambiguous` 全删、按 `reject` 全删，而是分层处理：

- `clean_*`: 直接保留
- `ambiguous_other_target / ambiguous_target_overlap`: 最优先人工复核，强烈怀疑错类
- `reject_positive_miss`: 重点复核，必要时剔除
- `ambiguous_extra_context`: 视任务目标决定是重切分还是保留
- `reject_negative_exact_target`: 从负样本中优先拿出来人工确认
- `reject_load_error`: 单独排查音频文件完整性和读取问题

也就是说，这次 ASR 筛洗更适合作为“人工复核前的预分桶”，而不是最终清洗结论本身。
