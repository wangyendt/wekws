## 四、当前阶段结论

1. 在当前已完成的小雷原始 test split 对比里，top20 weight surgery `159.pt` 仍是最稳的老师：`小雷小雷 99.25%`、`小雷快拍 99.87%`。
2. 首轮 `distill 199K 119.pt` 已明显掉点：`小雷小雷` 从老师的 `99.25%` 降到 `96.70%`，`小雷快拍` 从 `99.87%` 降到 `99.61%`。
3. `avg_ft_20.pt` 没有把首轮蒸馏救回来，反而进一步降到 `小雷小雷 94.23%`、`小雷快拍 98.95%`。
4. 第二轮长 schedule 的 `distill 199K tuned 299.pt` 也没有改善，反而比首轮 `119.pt` 更差：`小雷小雷 96.70% -> 94.53%`，`小雷快拍 99.61% -> 98.95%`。
5. 回归嗨小问 continuation 思路后的 `distill 199K replay_from79 229.pt` 比 `299.pt tuned` 有所回升，但仍未超过首轮 `119.pt`：`小雷小雷 96.48% < 96.70%`，`小雷快拍 99.35% < 99.61%`。
6. `distill 199K replay_from99 229.pt` 已补完，但结果明显更差：`小雷小雷 91.53%`、`小雷快拍 96.99%`。它不仅没有超过 `119.pt`，还明显落后于 `replay_from79`，因此这条 continuation seed 可以直接判失败。
7. 同样的 fixed-lr continuation 如果从 `119.pt` 继续，结果也更差：`distill 199K replay_from119 229.pt` 只有 `小雷小雷 93.78%`、`小雷快拍 98.04%`，说明“从更晚 checkpoint 继续训”并没有自动变好，反而可能把已有的 `CTC` 路径进一步冲坏。
8. 从 `119.pt` 直接继续 finetune 的 `distill 199K ft_from119_ctc_v1 229.pt` 出现明显塌陷：`小雷小雷 84.18%`、`小雷快拍 98.30%`。这已经不是“小退一点”，而是当前 from119 continuation 家族里最差的一条。
9. 到 `replay_from79 / 99 / 119` 三个 seed 全部补齐后，可以确认：当前 `FA + CTC continuation` 路线没有任何一条超过首轮 `119.pt`。`replay_from79` 是最接近的一条，但仍然略差；`replay_from99`、`replay_from119` 和 `ft_from119_ctc_v1` 都属于明显负收益分支。
10. 第一版输出蒸馏 `distill 199K output_kd_from119_v1 avg_30.pt` 没有通过原始 `test split` gate：`小雷小雷 92.35%`、`小雷快拍 99.22%`。它虽然比 `replay_from99` 和 `ft_from119_ctc_v1` 好，但仍明显低于 `119.pt` 和 `replay_from79`，说明“只加 output KL + blank/non-blank KD”还不足以把召回拉回当前最好学生。
11. 这版输出蒸馏在两项关键词上都选到了 `threshold=0.000`，且 `小雷小雷` 的 `fa_per_hour=0.00`。这更像模型整体变得偏保守，而不是单纯阈值没选好；就原始 `test split` 而言，它没有把 recall 问题解决掉。
12. 因为 `replay_from99` 没有通过“超过 `119.pt`”这个 gate，所以当前仍不补跑 199K continuation 家族的外部 `test.list`；现阶段这条线应阶段性收口，转向针对漏检模式的训练干预。
13. 在 `replay_from79/test_229` 的 `小雷小雷` 漏召回里，共有 47 条正样本未命中；其中 392K 老师 `159.pt` 能召回 39 条，仅有 8 条连老师也一起漏掉。
14. 代表样本 `XIAOLEI_XIAOLEI_0031_1m_f_04_fast_01_rep_10` 的单条诊断说明：两个 199K 学生都塌成了 `雷 小 雷`，而 392K 老师能形成稳定的 `小 雷 小 雷` 完整路径并以 `0.999` 触发，因此当前主要矛盾不是阈值，而是学生没有保住非 blank 激活和关键词内部时序结构。
15. 在 `119.pt` 的 10 条“老师收、学生漏”批量 `logit / prefix` 图里，失败模式可粗分为三类：重复扩展失败 `4/10`、前缀起步失败 `3/10`、老师自己也不完全稳定 `3/10`。当前更像“前缀起步偏弱 + 重复结构扩展失败”，而不只是“第一个小概率低”。
16. 把这 10 条代表性漏检样本都做成 `x3` 流式 repeated hard case 后，旧学生 `119.pt` 的结果是 `0/10` 能做到真正的 `3/3`；`10/10` 都稳定漏掉第一段 `segment 0`。这说明问题不是随机漏检，而是部署相关的早期 prefix / 第一段起步失守。
17. 在外部 test.list 上，按统一口径 `FA/h<=1.0` 比较，baseline2599 `79.pt` 为 `74.35%`，top20 surgery `159.pt` 提升到 `91.57%`。
18. 如果不重新在外部集调阈值，而是直接沿用原始 test split 上的阈值，则两版老师模型在外部集上都能保持约 `98%` 召回，但误唤醒明显偏高：baseline `54/763`，top20 surgery `50/763`。
19. 用户之前记住的“约 96% 召回”对应的是 baseline2599 在外部集更宽松的高误唤醒工作点，不是当前这份总表统一采用的 `1 FA/h` 工作点。
20. 第一版 `hard positive replay` 先在 train split 上挖出了 `826` 条 `teacher-hit / student-miss` 的 `小雷小雷` hard positives，`replay_factor=3` 后仅额外增加了 `2478` 条训练样本，占总 train 的约 `1.55%`。这说明首轮 replay 干预其实偏弱。
21. `distill 199K hardpos_replay_from119_v1 avg_30.pt` 的原始 test split 结果为 `小雷小雷 93.48%`、`小雷快拍 99.35%`。它比 `output_kd_from119_v1` 有所回升，但仍明显低于当前最好学生 `119.pt (96.70% / 99.61%)`，因此主 gate 仍未通过。
22. 这版 replay-only v1 的全量 streaming 统计与原始 test split 完全一致，没有出现“静态一般但 streaming 变好”的情况。
23. 在两条最关键的 repeated hard case `0031/0021` 上，`hardpos_replay_from119_v1` 都只有 `1/3`，而且都只触发 `segment 1`；相比之下，旧学生 `119.pt` 至少还能做到 `2/3`、覆盖 `segment 1/2`。这说明 replay-only v1 没有救回部署相关的 prefix/path 问题，反而在代表性 hard case 上更差。
24. 结合 `826` 条 hard positive 只额外回放了 `2478` 条这一事实，更合理的解释是：首轮 replay 信号太弱，而 `from119` 的长 schedule finetune 漂移仍然过强。下一步如果还继续 replay 路线，应优先做“更强 replay + 更短 schedule”，而不是继续维持当前 `replay_factor=3 + finetune 60 epoch`。
25. 第二版 `hard positive replay` 已经直接验证了上面这个假设：把 `replay_factor` 从 `3` 提到 `10`、把 `finetune_epochs` 从 `60` 缩到 `30`、把 `finetune_lr` 从 `1e-4` 降到 `5e-5` 后，`小雷小雷` 也只从 `93.48%` 微涨到 `93.55%`，而 `小雷快拍` 反而从 `99.35%` 掉到 `99.09%`。这说明“更强 replay + 更短 schedule”并没有带来实质改善。
26. `hardpos_replay_from119_v2` 的全量 streaming 结果与静态仍然完全一致，也没有出现隐藏的流式收益。
27. 更关键的是，两条代表性 repeated hard case `0031/0021` 在 `v2` 上仍然都是 `1/3`，且都只触发 `segment 1`；对比旧学生 `119.pt` 的 `2/3`，说明 replay-only 这条线到 `v2` 仍然没有修复 prefix/path 的部署问题。
28. 因为 `v1 -> v2` 已经同时验证了两件事都不成立：
    - 不是单纯 replay 太弱
    - 也不是单纯 schedule 太长
    所以下一步不建议继续堆 `replay_factor` 或继续做 `v3/v4`。更合理的下一步应该切到 `teacher-hit / student-miss` 样本加权，而不是继续 replay-only。
29. 第一版样本加权 `hardpos_weight_from119_v1` 直接把 `826` 条 hard positive 设成 `hard_weight=5.0`，并对 `CTC/MSE/KD` 统一做 all-loss weighting，结果明显变差：`小雷小雷` 从 `93.55%` 掉到 `90.85%`，`小雷快拍` 从 `99.09%` 掉到 `98.43%`。
30. `hardpos_weight_from119_v1` 的全量 streaming 结果与静态仍然完全一致，没有出现任何隐藏的流式收益。
31. 更关键的是，两条代表性 repeated hard case `0031/0021` 在 `hardpos_weight_from119_v1` 上仍然都只有 `1/3`，且都只触发 `segment 1`，没有恢复到旧学生 `119.pt` 的 `2/3`。
32. 这说明样本加权这条线并非一定错误，但当前这版 **all-loss weighting** 明显把模型带偏了。下一步更合理的方向不是继续堆 `hard_weight=8/10`，而是先把加权范围收窄到 `CTC only`，避免把 `MSE/KD` 分支也一起放大。
33. 第二版 `hardpos_weight_ctc_only_from119_v1` 把加权范围收窄到 `CTC only` 后，`avg_30.pt` 也只有 `小雷小雷 91.00%`、`小雷快拍 98.43%`。它只比 all-loss weighting 的 `90.85% / 98.43%` 略好一点，但仍明显低于 replay-only v1/v2，更远低于 `119.pt`。
34. 这版 `CTC only weighting` 的 streaming 结果与静态完全一致，也没有带来任何隐藏的流式收益。
35. 两条代表性 repeated hard case `0031/0021` 在 `CTC only weighting` 上仍然都是 `1/3`，且都只触发 `segment 1`；没有恢复到旧学生 `119.pt` 的 `2/3`，也没有优于 replay-only v1/v2。
36. 额外检查早期 checkpoint 后，`124.pt` 只有 `93.03% / 98.82%`，`129.pt` 更差到 `90.40% / 98.30%`。这说明问题不只是“avg_30 混入了后期漂移”，而是样本加权这条线本身就没有把模型拉回正确方向。下一步更合理的是停止 sample-weighting，转成只动输出头的更克制微调。
37. 第一版 `head-only hardpos replay` 明显是当前为止最有效的 199K 修复路线：`avg_10.pt` 直接把静态拉到 `98.20% / 99.74%`，不仅大幅超过 `119.pt (96.70% / 99.61%)`，也首次逼近老师 `159.pt (99.25% / 99.87%)`。
38. 这版的全量 streaming 结果与静态完全一致，说明提升不是“只在非流式评测上好看”，而是真正能保留到部署口径。
39. repeated hard case 上，这版不再是全面卡死：`0021 x3` 已经提升到学生 `3/3`，说明冻结 backbone、只修 HEAD 路径这条思路确实能修复一部分 prefix/path 问题。
40. 但 `0031 x3` 在 `avg_10.pt` 上仍然只有 `1/3`，说明这条线虽然第一次真正有效，但还没完全收敛。
41. 同一条 run 内部还出现了一个重要 tradeoff：`129.pt` 的静态略低于 `avg_10.pt (98.13% / 99.74%)`，但 `0031 x3` 从 `1/3` 提升到了 `2/3`，而 `0021 x3` 仍保持 `3/3`。这说明“更晚、更尖锐的单 checkpoint”比平均模型更有 hardcase 恢复能力。
42. 基于上面这个现象，下一步最合理的不是回到样本加权，而是继续沿 `head-only` 方向做第二版：延长少量 head-only finetune，重点观察更晚 checkpoint 是否能把 `0031` 继续从 `2/3` 拉到 `3/3`，同时尽量保持静态精度不明显回落。
43. `head-only hardpos replay v2` 的 `avg_5.pt` 与 `v1 avg_10.pt` 在静态和全量 streaming 上完全相同，都是 `98.20% / 99.74%`。这说明单纯把 `head-only` finetune 从 `10` 拉到 `20` epoch，没有带来任何新增静态收益。
44. 更晚的 `139.pt` 也没有复现 `v1/129.pt` 那种 hardcase tradeoff：它的静态仍是 `98.20% / 99.74%`，但 `0031 x3` 依旧只有 `1/3`，而 `0021 x3` 维持 `3/3`。这说明继续做 epoch/checkpoint sweep 已经基本进入平台期。
45. 因此下一步不应再继续拉长 `head-only` schedule，而应把 `826` 条 hard positives 再分桶，优先只强化“前缀起步失败 / 雷小雷塌缩”子集，看看能否更有针对性地推动 `0031` 这类样本。
46. `prefixstart_replay_head_only_from119_v1` 从 `826` 条 hard positives 中筛出了 `513` 条 `greedy_not_start` 子集，主失败模式是 `雷 小 雷`，再以 `replay_factor=20` 生成训练集；但 `avg_10.pt` 只有 `98.05% / 99.74%`，低于 `head-only hardpos replay v1/v2` 的 `98.20% / 99.74%`。
47. 这版的全量 streaming 与静态一致，也是 `98.05% / 99.74%`，说明 prefix-start 子集 replay 没有带来隐藏流式收益。
48. hardcase gate 也没有通过：`0031 x3` 仍只有 `1/3`，`0021 x3` 保持 `3/3`；更晚的 `129.pt` 静态仍是 `98.05% / 99.74%`，`0031 x3` 也仍只有 `1/3`。因此“只收窄到 prefix-start 子集并强 replay”不是下一步主线。
49. 额外做了 `0031 x3` 的 `gap=300ms` 诊断：老师仍然 `3/3`，最佳 head-only 线的 `avg_10.pt` 仍只有 `1/3`，同 run 的 `129.pt` 变为 `2/3` 但仍不到 `3/3`。这说明问题不只是零间隔拼接或 decoder reset，而是学生路径本身仍没有稳定学会连续多次 `小雷小雷`。
50. 下一步更合理的是构造 repeated-sequence 训练样本：把 train 中的 hard positive 拼成 `小雷小雷 x3` 长音频，同时把 CTC 标签也扩展成三遍 `小 雷 小 雷`，让训练目标直接约束连续多次触发，而不是继续重复短样本。
