# 20260423 199K 样本加权 CTC-only v1 实施方案

## 1. 为什么从 all-loss weighting 改成 CTC-only

`hardpos_weight_from119_v1` 已经证明：

- 直接把 `826` 条 hard positives 设成 `hard_weight=5.0`
- 并且对 `CTC / MSE / KD` 统一做 all-loss weighting

会明显把模型带坏。

最像的问题不是：

- hard positives 这条思路本身错了

而是：

- 不应该把 hard weight 同时放大到 `MSE/KD` 分支

因为当前最需要被强化的是：

- 学生对 `小雷小雷` 的识别路径

这本质上更接近：

- `CTC` 层面的识别问题

而不是：

- block feature alignment 本身不够强

## 2. 本轮核心假设

这轮只验证一个更窄的假设：

- 仍然保留同一批 `826` 条 hard positives
- 仍然保留 `hard_weight = 5.0`
- 但 sample weighting 只作用在 `CTC` 分支

也就是：

- `sample_weight_scope = ctc_only`

如果这轮比 all-loss weighting 明显回升，就说明：

- 问题不是 hard positive weighting 这条思路本身
- 而是前一轮把加权范围施加得过宽

## 3. 实验设置

### 3.1 起点

仍然从：

- `exp/fsmn_ctc_xlxl_distill_199k/119.pt`

起步，不从 `hardpos_weight_from119_v1` 继续接着训。

### 3.2 训练参数

- `hard_weight = 5.0`
- `default_weight = 1.0`
- `sample_weight_scope = ctc_only`
- `align_epochs = 120`
- `finetune_epochs = 30`
- `finetune_lr = 5e-5`
- `head_lr_ratio = 0.1`
- `finetune_mse_weight_start = 0.3`
- `finetune_mse_weight_end = 0.1`
- 不叠 `output KD`

## 4. 代码实现原则

这轮不再修改数据集本身，而是：

1. 继续使用同一个 `key -> weight` JSON 映射
2. 在训练参数中加入：
   - `sample_weight_scope`
3. 首轮只允许：
   - `CTC` 使用 sample weight
4. `MSE / KD / blank-KD` 保持普通 batch mean

## 5. Gate

### 5.1 基本 gate

至少要满足：

- `小雷小雷` 明显高于 `all-loss weighting v1` 的 `90.85%`
- `小雷快拍` 不能继续掉

### 5.2 真实 gate

真正决定是否继续沿这条线走的，还是这两件事：

1. 全量 streaming
2. repeated hard case `0031 / 0021`

如果这轮还是：

- `1/3`

那说明单独把加权范围收窄到 `CTC only` 仍然不够。

## 6. 可能的下一步分叉

### 6.1 如果 CTC-only v1 有明显回升

下一步可以继续小范围探索：

- `hard_weight = 3.0`
- `hard_weight = 5.0 + light output KD`

### 6.2 如果 CTC-only v1 仍然很差

那就说明：

- 单独 sample weighting 这条线整体价值有限

下一步应转向：

- `sample weighting + output KD`

或者更直接地：

- prefix/path 约束
