# 20260403 小雷 199K 蒸馏方案

## 1. 结论

下一步应当先复用嗨小问现成的 `199K` 蒸馏路线，把当前已经验证通过的 `top20 weight surgery` 老师模型继续压到约 `199K`。

当前建议的老师/学生组合：

- 老师模型：`exp/fsmn_ctc_xlxl_top20_weight_surgery/159.pt`
- 老师词表：`dict_top20_xlxl`
- 学生配置：`conf/fsmn_ctc_student_mini.yaml`
- 目标实验目录：`exp/fsmn_ctc_xlxl_distill_199k`

原因：

- `top20 weight surgery 159.pt` 已经在原始 `test split` 和外部 `test.list` 上完成验证。
- 外部 `test.list` 上，在统一 `1 FA/h` 口径下，`top20 surgery 159.pt` 明显优于 `baseline2599 79.pt`。
- 现在 head 已经稳定为 `top20`，继续压缩的主要目标应当是 `backbone`，而不是再折腾词表。
- `fsmn_ctc_student_mini.yaml` 就是仓库里现成的 `199K` 路线，后续继续做 `S3` 蒸馏时也能复用同一套 `dict_top20_xlxl`。

## 2. 复用哪条现成链路

本次直接复用仓库中已经实现好的 Feature Alignment 蒸馏链路：

- 训练脚本：`examples/hi_xiaowen/s0/run_distill.sh`
- 训练主程序：`wekws/bin/train_distill.py`
- 蒸馏执行器：`wekws/utils/executor_distill.py`
- 学生配置：`examples/hi_xiaowen/s0/conf/fsmn_ctc_student_mini.yaml`

这条链路不是传统 logits KD，而是：

1. Phase 1：学生 backbone 对齐老师 block-level feature（纯 MSE）
2. Phase 2：MSE + CTC 混合微调，HEAD 低学习率解冻

对小雷来说，这条路是兼容的，因为：

- 老师已经是 `top20` 输出空间
- 学生从第一天就使用 `dict_top20_xlxl`
- 后续评测、导出、量化都继续沿用 `top20` 词表，不会再引入新一套 dict

## 3. 目标模型与尺寸

学生配置 `conf/fsmn_ctc_student_mini.yaml` 的设计目标就是约 `199K`：

- 教师 backbone：约 `390K`
- 教师 top20 surgery 总参数：`392,494`
- 学生 mini backbone：约 `197K`
- 学生 20 类总参数：约 `199K`

因此本轮实验的核心目标是：

- 在原始 `test split` 上尽量保持当前 `top20 surgery` 精度
- 在外部 `test.list` 上尽量保持当前低误唤醒约束下的优势
- 先拿到一版稳定的 `199K` 学生，再决定是否继续压到 `S3`

## 4. 训练配置

本轮建议固定如下：

- 数据目录：`data_xlxl_0327_ctc_v1_clean`
- 老师 checkpoint：`exp/fsmn_ctc_xlxl_top20_weight_surgery/159.pt`
- 词表目录：`dict_top20_xlxl`
- 输出类别数：`20`
- 学生配置：`conf/fsmn_ctc_student_mini.yaml`
- 训练 GPU：`0,1,2,3`

蒸馏 phase：

- `align_epochs=100`
- `finetune_epochs=20`
- 总 epoch：`120`
- 预期最终 checkpoint：`119.pt`

默认先不改这些超参，先严格复用嗨小问 199K 路线；如果首版结果不理想，再考虑第二轮调参。

## 6. 验收标准

第一轮 199K 蒸馏，建议优先看这几个点：

1. 原始 `test split` 上，不能明显劣于当前 `top20 surgery 159.pt`。
2. 外部 `test.list` 上，在统一 `1 FA/h` 口径下，尽量保持当前 `top20 surgery` 的优势。
3. 如果 `119.pt` 和 `avg_30.pt` 接近，则优先保留 `avg_30.pt` 作为后续导出和更小模型蒸馏的老师。

## 7. 之后的路线

如果这版 `199K` 结果稳定，下一步再进入更小学生，比如 `S3`：

- 老师：优先考虑 `199K avg_30.pt`
- 词表：继续固定 `dict_top20_xlxl`
- 评测：继续沿用当前三张总表的统一口径

这样后面做 `S3` 蒸馏、量化、LiteRT/ExecuTorch 导出时，路径会比直接从 `392K` 老师跳更小模型更稳。


## 补充：首轮 199K 掉点后的判断与第二轮建议

首轮 exp/fsmn_ctc_xlxl_distill_199k 的 119.pt 已经完成评测，原始 test split 上结果为：小雷小雷 96.70%，小雷快拍 99.61%。相对老师 top20 weight surgery 159.pt，小雷小雷从 99.25% 下降到 96.70%，小雷快拍从 99.87% 下降到 99.61%，说明这版 199K 还不能视为几乎无损。

当前 avg_30 结果更差，不能直接作为 199K 的有效结论。原因是 average_model.py 在 --val_best 下只按 yaml 里的 cv_loss 选 checkpoint，而 distill 两阶段训练的 cv_loss 不能跨 phase 直接比较。实际被选进去的大量 checkpoint 来自 align phase，而不是 finetune 后段，因此 avg_30 混入了 HEAD 还没真正完成 CTC 微调的模型。

当前更合理的 follow-up 有两个：

1. 先只平均 finetune 阶段 100~119，生成 avg_ft_20，再评测它。
2. 如果 119.pt 和 avg_ft_20 仍明显低于老师，直接做第二轮长 schedule，优先改成 align_epochs=200、finetune_epochs=100、head_lr_ratio=0.1。

这样做的理由是：当前 recipe 比历史强配置更短，HEAD 学习率也更保守，而且 finetune 末尾验证指标仍在改善，说明训练结束时不像完全收敛。第二轮应该先拉长训练，而不是先怀疑评测脚本或实现 bug。
