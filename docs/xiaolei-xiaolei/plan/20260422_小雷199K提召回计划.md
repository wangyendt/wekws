# 20260422 小雷 199K 提召回计划

## 1. 这份计划回答什么问题

`replay_from99` 已经跑完，而且没有超过 `119.pt`。

因此今天这份计划只回答一个问题：

- 接下来应该怎么做，才能提高 `199K` 学生模型的召回率

这里的“提高召回率”默认优先指：

- `小雷小雷`

同时要求：

- 不把 `小雷快拍` 一起明显拉坏
- 不靠简单放宽阈值来制造表面提升

## 2. 当前阶段结论

截至今天，关键模型结果如下：

| 模型 | checkpoint | 小雷小雷 | 小雷快拍 | 结论 |
|---|---|---:|---:|---|
| `top20 weight surgery` | `159.pt` | 99.25% | 99.87% | 当前老师 |
| `distill 199K` | `119.pt` | 96.70% | 99.61% | 当前最好学生 |
| `distill 199K replay_from79` | `229.pt` | 96.48% | 99.35% | 接近 `119.pt`，但仍略差 |
| `distill 199K replay_from99` | `229.pt` | 91.53% | 96.99% | 明显更差 |
| `distill 199K replay_from119` | `229.pt` | 93.78% | 98.04% | 更差 |
| `distill 199K ft_from119_ctc_v1` | `229.pt` | 84.18% | 98.30% | 明显跑坏 |

这意味着：

1. `79 / 99 / 119` continuation seed sweep 已全部补齐
2. 没有任何 continuation 分支超过 `119.pt`
3. 当前 `FA + CTC continuation` 路线阶段性收口
4. 后续不要再扩新的 from99 / from119 / `v2` 类变体

## 3. 当前对失败模式的判断

基于已有批量 `logit / prob / prefix` 图和单条诊断，当前更可靠的判断是：

1. 问题不只是“第一个 `小` 概率低”
2. 更像“前缀起步偏弱 + 重复结构扩展失败”
3. 学生常见退化输出是：
   - `雷 小 雷`
   - `小 雷`
   - `小 小 雷`
4. 老师更常见的是：
   - `<filler> 小 雷 小 雷`

所以当前真正要救的不是某一个单独 token，而是：

- `blank / filler -> 小1` 的前缀起步
- `小雷 -> 小雷小 -> 小雷小雷` 的路径保持

## 4. 下一步主线

### 4.1 基线固定

后续所有提召回实验，先统一以 `119.pt` 作为当前最好学生基线。

原因：

- 它仍然是所有 `199K` 学生里最好的
- `replay_from79` 虽接近，但没有超过
- `replay_from99 / 119 / ft_from119_ctc_v1` 都已经说明继续沿现有 continuation 走，收益很差

### 4.2 第一优先级：hard positive replay

这是下一步最该先做的训练改动。

目标样本：

- 老师 `159.pt` 能稳定命中
- 学生 `119.pt` 漏掉
- 关键词优先 `小雷小雷`

清单建议分三桶：

1. 前缀起步失败
   - 典型输出：`雷 小 雷`
2. 重复扩展失败
   - 典型输出：`小 雷`、`小 小 雷`
3. 老师也不稳定
   - 老师自己也只到 `<filler> 小 雷` 或 `<filler> 小 雷 小`
   - 这桶先不拿来做强监督

训练方式建议：

- 仍然保留原 clean 训练集主体
- 额外把前两桶 hard positive 重复采样或复制若干倍
- 先做一版最轻量的数据层干预，不急着先改 loss

### 4.3 第二优先级：teacher-hit / student-miss 样本加权

如果 hard positive replay 还不够，再进入样本加权。

更合理的加权对象不是单个 token，而是：

- 老师稳定命中、学生漏掉的整条样本

更合理的加权目标也不是“给 `小` 加权”，而是：

- 强化前缀起步
- 强化重复结构扩展

原因：

- 当前真正坏掉的是 prefix/path
- 不是简单某个 `小` 字在所有位置都不够高

### 4.4 第三优先级：prefix / path 约束

只有前两步还不够，再考虑 loss 级改动。

更值得约束的是这四条 prefix：

- `小`
- `小雷`
- `小雷小`
- `小雷小雷`

而不是直接围绕单个 `小 / 雷` token 做全局加权。

## 5. 数据增强放在哪

当前不把泛化数据增强放第一优先级。

原因：

- 同一批数据上老师已经能收
- 说明样本本身不是完全缺失
- 问题更像学生没有把正确前缀路径学稳

如果要做增强，也只做定向增强，而且放在 hard positive replay 之后：

1. 随机前导 `filler / silence / noise`
2. 随机平移关键词在 `2s` 窗口里的位置
3. 强化 `<filler> -> 小1` 这段过渡

当前不建议作为第一反应的方向：

- 先上泛化噪声增强
- 先上更重的 SpecAug
- 直接做“给 `小` 单字加权”

## 6. 执行顺序

当前最合理的顺序是：

1. 先用 `119.pt` 挖 hard positive 清单
2. 先做一轮“重复 hard case 的流式稳定性实验”，确认老师/学生在长音频串联场景下的多次触发行为
3. 重点看三件事：
   - 老师在重复拼接后能否稳定触发多次
   - 学生是否在长音频里更容易丢前缀或漏掉后续触发
   - 整段 greedy / beam 里 `<filler>` 是否在每次关键词前都自然出现，还是只是单条短音频上的局部现象
4. 如果老师在串联场景下也不稳，则不能把“老师短音频上的 `<filler>` 过渡”直接当成必须强蒸馏的目标
5. 如果老师在串联场景下稳定，而学生不稳定，则更支持后续做 output-level KD / blank-vs-nonblank KD / hard positive replay
6. 完成流式稳定性检查后，再把样本按“前缀起步失败 / 重复扩展失败 / 老师不稳”分桶
7. 先做一版 hard positive replay
8. 只要这版超过 `119.pt`，再补跑外部 `test.list`
9. 如果 hard positive replay 仍不够，再上样本加权
10. 只有前两步都不够，再上 prefix/path 约束

### 6.1 重复 hard case 的流式稳定性实验

这一步的目标不是直接提召回，而是先回答一个部署相关问题：

- 如果把同一条 hard case 连续拼接成更长音频，老师和学生在流式推理下会发生什么

建议先从最典型的 `小雷小雷` hard case 开始，做：

1. 原始 `3s` 音频重复拼接成 `9s`
2. 同时看 teacher / student
3. 同时看非流式与流式

重点观察：

- 流式模式下是否能稳定打出 `3` 次触发
- 每次触发的时间位置是否落在对应段内
- 整段 greedy / beam 是否出现多次 `<filler>`
- `<filler>` 在长音频里到底是稳定过渡状态，还是单条短音频里的偶发现象

这一步的结论会直接影响后面的训练设计：

- 如果老师在长音频里本来就不会每次都出现 `<filler>`，那后续训练重点就不该是“逼学生学会固定的 `<filler>` 峰”
- 如果老师稳定多次触发，而学生明显少触发或只在第一段触发，那就更支持从训练目标里强化 prefix/path 保持

## 7. 当前明确不做的事

- 不再继续扩新的 continuation seed
- 不再继续做 from119 的 `v2` 类变体
- 不把主要精力放在扫阈值上
- 不把“给 `小` 单字加权”当成第一反应
